# Требования к серверу для HT

## Стек

| Сервис | Описание | Назначение |
|--------|----------|------------|
| **api** | FastAPI + Uvicorn | Админка API, контент-план, загрузки |
| **bot** | aiogram | Telegram-бот, polling |
| **db** | PostgreSQL 15 | База данных |
| **minio** | MinIO | S3-хранилище для медиа (опционально) |
| **frontend** | nginx + React | Статика админки |

### Медиа на Timeweb S3 (s3.twcstorage.ru)

Чтобы хранить загрузки и отдавать медиа с **https://s3.twcstorage.ru**, в `.env` на сервере задайте:

```bash
S3_ENDPOINT_URL=https://s3.twcstorage.ru
S3_BUCKET=имя-бакета
S3_ACCESS_KEY_ID=...
S3_SECRET_ACCESS_KEY=...
S3_REGION=ru-1
S3_PUBLIC_BASE_URL=https://s3.twcstorage.ru/имя-бакета
```

Деплой при наличии `S3_ENDPOINT_URL` поднимет конфигурацию без MinIO; API и бот будут брать медиа по этому URL.

## Рекомендуемые характеристики

### Минимум (до ~500 пользователей, низкая активность)

| Ресурс | Значение | Комментарий |
|--------|----------|-------------|
| **CPU** | 1 vCPU | Может быть 100% при рассылке; возможны задержки |
| **RAM** | 1.5 GB | ~600 MB контейнеры + запас |
| **Диск** | 20 GB SSD | Образы ~3 GB, БД, uploads |
| **Сеть** | 100 Mbit/s | Достаточно для типичной нагрузки |

### Оптимально (до ~5000 пользователей, стабильная работа)

| Ресурс | Значение | Комментарий |
|--------|----------|-------------|
| **CPU** | 2 vCPU | Запас при рассылках и пиках |
| **RAM** | 2–4 GB | Комфортный запас для всех сервисов |
| **Диск** | 40–60 GB SSD | Рост БД и медиа |
| **Сеть** | 100 Mbit/s | Обычно хватает |

### Высокая нагрузка (много пользователей, частые рассылки)

| Ресурс | Значение |
|--------|----------|
| **CPU** | 4 vCPU |
| **RAM** | 4–8 GB |
| **Диск** | 80+ GB SSD |

## Оценка потребления по контейнерам

| Контейнер | RAM (idle) | RAM (нагрузка) | CPU |
|-----------|------------|----------------|-----|
| PostgreSQL | 150–250 MB | 300–500 MB | низкая |
| MinIO | 100–150 MB | 200–400 MB | низкая |
| API | 100–150 MB | 200–400 MB | средняя |
| Bot | 80–120 MB | 150–300 MB | низкая |
| Frontend (nginx) | 10–30 MB | 20–50 MB | минимальная |
| **Итого** | ~500–700 MB | ~1–1.5 GB | |

---

## Что влияет на нагрузку

1. **Рассылки** — отправка в бот и каналы (HTTP к Telegram API). При 1000+ пользователях — заметный всплеск CPU и сети.
2. **Загрузка медиа** — MinIO и диск при больших файлах.
3. **Параллельные запросы к API** — несколько админов, аналитика.
4. **Размер БД** — логи, история рассылок, пользователи.

---

## Рекомендации по оптимизации

- **RAM**: 2 GB — разумный минимум для стабильной работы.
- **SSD** — обязателен, HDD даёт задержки на I/O.
- **Swap** — 1–2 GB при 1–2 GB RAM снижает риск OOM.
- **Один сервер** — для текущего масштаба достаточно одного VPS.

## Деплой через GitHub Actions (CD)

Чтобы CD по push в `main` подключался по SSH к серверу, **порт 22 должен быть доступен** с IP GitHub Actions (или с интернета, если используется только ключ).

На сервере один раз выполните:

```bash
sudo ./deploy/firewall-github-actions.sh
```

Скрипт добавит в UFW правила для IP-диапазонов GitHub Actions. Если используете не UFW (например, iptables или файрвол облака), откройте порт 22 для входящих с [GitHub meta API](https://api.github.com/meta) → `actions`.

В репозитории заданы секреты: `SSH_HOST`, `SSH_USER`, `SSH_PRIVATE_KEY`.

Если видите ошибку `dial tcp ...:22: i/o timeout` — runner не достучивается до сервера: проверьте файрвол, что SSH слушает на 22 и хост доступен из интернета.
